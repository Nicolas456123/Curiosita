01

Introduction

Stocker et interroger les données

Une base de données est un système organisé pour stocker, gérer et récupérer des données de manière efficace et fiable. Toute application web qui gère des utilisateurs, des produits, des articles ou des transactions repose sur une base de données. Comprendre les bases de données est essentiel pour concevoir des applications performantes et robustes.

Deux grandes familles : les bases relationnelles (SQL) organisent les données en tables avec des relations (PostgreSQL, MySQL, SQLite), tandis que les bases NoSQL utilisent des modèles flexibles — documents (MongoDB), clé-valeur (Redis), colonnes (Cassandra) ou graphes (Neo4j).

Le choix entre SQL et NoSQL dépend du type de données, des besoins de cohérence, de la charge et de la complexité des requêtes. De nombreuses applications modernes combinent les deux approches selon les cas d'usage.

Le cours

SQL, NoSQL et modélisation

Modélisation relationnelle

La modélisation relationnelle organise les données en tables (ou relations). Chaque table a des colonnes (attributs) et des lignes (enregistrements). Les relations entre tables sont établies par des clés primaires (identifiant unique) et des clés étrangères (référence vers une autre table).

Types de relations :
Un-à-un (1:1) — un utilisateur a un profil
Un-à-plusieurs (1:N) — un auteur écrit plusieurs articles
Plusieurs-à-plusieurs (N:N) — des étudiants suivent plusieurs cours, et chaque cours a plusieurs étudiants (table de jonction nécessaire)

Normalisation : processus d'élimination des redondances en décomposant les tables. Les trois premières formes normales (1NF, 2NF, 3NF) suffisent dans la plupart des cas.

SQL : créer et structurer

SQL (Structured Query Language) est le langage standard pour interagir avec les bases relationnelles. Le sous-ensemble DDL (Data Definition Language) sert à créer et modifier la structure des tables.

SQL
-- Créer une table
CREATE TABLE utilisateurs (
  id        SERIAL PRIMARY KEY,
  nom       VARCHAR(100) NOT NULL,
  email     VARCHAR(255) UNIQUE NOT NULL,
  age       INTEGER CHECK (age >= 0),
  cree_le   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Table avec clé étrangère
CREATE TABLE articles (
  id          SERIAL PRIMARY KEY,
  titre       VARCHAR(200) NOT NULL,
  contenu     TEXT,
  auteur_id   INTEGER REFERENCES utilisateurs(id),
  publie_le   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Table de jonction (relation N:N)
CREATE TABLE articles_tags (
  article_id  INTEGER REFERENCES articles(id),
  tag_id      INTEGER REFERENCES tags(id),
  PRIMARY KEY (article_id, tag_id)
);

SQL : interroger les données

Le sous-ensemble DML (Data Manipulation Language) permet d'insérer, lire, modifier et supprimer des données. La clause SELECT est la plus utilisée, avec WHERE pour filtrer, ORDER BY pour trier et GROUP BY pour agréger.

SQL
-- Insérer des données
INSERT INTO utilisateurs (nom, email, age)
VALUES ('Alice Dupont', 'alice@example.com', 28);

-- Lire avec filtres et tri
SELECT nom, email
FROM utilisateurs
WHERE age >= 18
ORDER BY nom ASC
LIMIT 10;

-- Agrégation
SELECT auteur_id, COUNT(*) AS nb_articles
FROM articles
GROUP BY auteur_id
HAVING COUNT(*) > 5;

-- Modifier
UPDATE utilisateurs SET age = 29 WHERE email = 'alice@example.com';

-- Supprimer
DELETE FROM articles WHERE id = 42;

Jointures SQL

Les jointures combinent les données de plusieurs tables en fonction d'une condition. C'est la puissance fondamentale du modèle relationnel : pouvoir reconstruire des vues complexes à partir de tables normalisées.

SQL
-- INNER JOIN : articles avec leur auteur
SELECT a.titre, u.nom AS auteur
FROM articles a
INNER JOIN utilisateurs u ON a.auteur_id = u.id;

-- LEFT JOIN : tous les utilisateurs, même sans article
SELECT u.nom, COUNT(a.id) AS nb_articles
FROM utilisateurs u
LEFT JOIN articles a ON u.id = a.auteur_id
GROUP BY u.nom;

-- Jointure multiple (N:N via table de jonction)
SELECT a.titre, t.nom AS tag
FROM articles a
JOIN articles_tags at ON a.id = at.article_id
JOIN tags t ON at.tag_id = t.id
WHERE t.nom = 'JavaScript';

Indexation et performances

Un index est une structure de données (souvent un B-tree) qui accélère la recherche dans une table, au prix d'un léger surcoût en écriture et en espace. Sans index, le moteur doit parcourir toute la table (full table scan).

SQL
-- Créer un index sur la colonne email
CREATE INDEX idx_users_email ON utilisateurs(email);

-- Index composé (pour les requêtes filtrant sur deux colonnes)
CREATE INDEX idx_articles_auteur_date
ON articles(auteur_id, publie_le DESC);

-- Analyser une requête avec EXPLAIN
EXPLAIN ANALYZE
SELECT * FROM utilisateurs WHERE email = 'alice@example.com';

Quand indexer ? Indexez les colonnes utilisées dans WHERE, JOIN, ORDER BY et GROUP BY fréquemment. Les clés primaires sont indexées automatiquement. Trop d'index ralentissent les écritures (INSERT, UPDATE, DELETE).

Bases NoSQL

Les bases NoSQL abandonnent le modèle tabulaire strict au profit de structures plus flexibles. MongoDB, la plus populaire, stocke des documents JSON dans des collections (équivalent des tables).

JavaScript (MongoDB avec Mongoose)
const mongoose = require('mongoose');

// Définir un schéma
const userSchema = new mongoose.Schema({
  nom:   { type: String, required: true },
  email: { type: String, unique: true, required: true },
  age:   Number,
  tags:  [String],       // tableau de chaînes
  adresse: {              // document imbriqué
    rue: String,
    ville: String
  }
});

const User = mongoose.model('User', userSchema);

// Créer
await User.create({ nom: 'Alice', email: 'alice@ex.com', age: 28 });

// Lire avec filtres
const users = await User.find({ age: { $gte: 18 } })
  .sort({ nom: 1 })
  .limit(10);

// Modifier
await User.updateOne(
  { email: 'alice@ex.com' },
  { $set: { age: 29 } }
);

SQL vs NoSQL : quand utiliser quoi ?

SQL (relationnel) :
- Données structurées avec des relations complexes
- Transactions ACID nécessaires (banque, e-commerce)
- Requêtes complexes avec jointures
- Schéma stable et bien défini

NoSQL (document, clé-valeur, etc.) :
- Données semi-structurées ou variables
- Scalabilité horizontale (gros volumes)
- Prototypage rapide (schéma flexible)
- Données imbriquées (profils, catalogues)

ACID vs BASE : SQL garantit Atomicité, Cohérence, Isolation, Durabilité. NoSQL privilégie souvent disponibilité et tolérance aux partitions (théorème CAP), avec cohérence « éventuelle ».

Question 1 / 6

Qu'est-ce qu'une clé primaire dans une base relationnelle ?

La clé primaire (PRIMARY KEY) identifie de manière unique chaque ligne d'une table. Elle ne peut pas être nulle ni dupliquée. Souvent un entier auto-incrémenté (SERIAL) ou un UUID.

Question 2 / 6

Quelle clause SQL permet de filtrer les résultats d'un SELECT ?

WHERE filtre les lignes selon une condition. HAVING sert à filtrer après un GROUP BY (sur les résultats agrégés).

Question 3 / 6

Quelle jointure retourne toutes les lignes de la table de gauche, même sans correspondance ?

LEFT JOIN retourne toutes les lignes de la table de gauche. Si aucune correspondance n'est trouvée dans la table de droite, les colonnes de droite contiennent NULL. INNER JOIN ne retourne que les correspondances.

Question 4 / 6

À quoi sert un index dans une base de données ?

Un index crée une structure de recherche (B-tree en général) qui permet de trouver rapidement les lignes correspondant à une condition, sans parcourir toute la table. C'est un compromis : lectures plus rapides, écritures légèrement plus lentes.

Question 5 / 6

Quelle est la principale caractéristique de MongoDB ?

MongoDB est une base de données orientée documents. Les données sont stockées au format BSON (Binary JSON) dans des collections. Le schéma est flexible : chaque document peut avoir une structure différente.

Question 6 / 6

Que signifie ACID dans le contexte des bases de données ?

ACID garantit la fiabilité des transactions : Atomicité (tout ou rien), Cohérence (les contraintes sont respectées), Isolation (les transactions concurrentes n'interfèrent pas), Durabilité (les données commitées survivent aux pannes).

Exercice 1 — Modéliser un blog

Concevez le schéma d'une base de données pour un blog. Vous devez avoir des utilisateurs, des articles, des commentaires et des catégories. Un article peut appartenir à plusieurs catégories. Écrivez les requêtes SQL de création des tables.

Voir la solution 
CREATE TABLE utilisateurs (
  id       SERIAL PRIMARY KEY,
  nom      VARCHAR(100) NOT NULL,
  email    VARCHAR(255) UNIQUE NOT NULL,
  cree_le  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE categories (
  id   SERIAL PRIMARY KEY,
  nom  VARCHAR(50) UNIQUE NOT NULL
);

CREATE TABLE articles (
  id         SERIAL PRIMARY KEY,
  titre      VARCHAR(200) NOT NULL,
  contenu    TEXT NOT NULL,
  auteur_id  INTEGER REFERENCES utilisateurs(id) ON DELETE CASCADE,
  publie_le  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE commentaires (
  id          SERIAL PRIMARY KEY,
  contenu     TEXT NOT NULL,
  auteur_id   INTEGER REFERENCES utilisateurs(id) ON DELETE CASCADE,
  article_id  INTEGER REFERENCES articles(id) ON DELETE CASCADE,
  cree_le     TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Relation N:N entre articles et catégories
CREATE TABLE articles_categories (
  article_id   INTEGER REFERENCES articles(id) ON DELETE CASCADE,
  categorie_id INTEGER REFERENCES categories(id) ON DELETE CASCADE,
  PRIMARY KEY (article_id, categorie_id)
);

Exercice 2 — Requêtes d'analyse

En utilisant le schéma du blog ci-dessus, écrivez les requêtes SQL pour : (1) les 5 articles les plus commentés, (2) le nombre d'articles par catégorie, (3) les utilisateurs n'ayant jamais écrit d'article.

Voir la solution 
-- 1. Les 5 articles les plus commentés
SELECT a.titre, COUNT(c.id) AS nb_commentaires
FROM articles a
LEFT JOIN commentaires c ON a.id = c.article_id
GROUP BY a.id, a.titre
ORDER BY nb_commentaires DESC
LIMIT 5;

-- 2. Nombre d'articles par catégorie
SELECT cat.nom, COUNT(ac.article_id) AS nb_articles
FROM categories cat
LEFT JOIN articles_categories ac ON cat.id = ac.categorie_id
GROUP BY cat.id, cat.nom
ORDER BY nb_articles DESC;

-- 3. Utilisateurs n'ayant jamais écrit d'article
SELECT u.nom, u.email
FROM utilisateurs u
LEFT JOIN articles a ON u.id = a.auteur_id
WHERE a.id IS NULL;

-- Alternative avec NOT EXISTS :
SELECT nom, email
FROM utilisateurs u
WHERE NOT EXISTS (
  SELECT 1 FROM articles a WHERE a.auteur_id = u.id
);

Exercice 3 — Requêtes paramétrées (sécurité)

Montrez la différence entre une requête vulnérable aux injections SQL et une requête paramétrée sécurisée. Utilisez Node.js avec le module pg (PostgreSQL).

Voir la solution 
const { Pool } = require('pg');
const pool = new Pool();

// DANGEREUX — injection SQL possible !
async function rechercheVulnerable(email) {
  // Si email = "' OR '1'='1" → toute la table est exposée
  const query = `SELECT * FROM utilisateurs WHERE email = '${email}'`;
  return pool.query(query);
}

// SÉCURISÉ — requête paramétrée
async function rechercheSecurisee(email) {
  const query = 'SELECT * FROM utilisateurs WHERE email = $1';
  return pool.query(query, [email]);
  // Le paramètre est échappé automatiquement par le driver
}

// Avec un ORM (Prisma) — sécurisé par défaut
const user = await prisma.utilisateur.findUnique({
  where: { email: email }
});

Exercice 4 — Optimisation avec index

Vous avez une table commandes avec 10 millions de lignes. Les requêtes fréquentes filtrent par client_id et trient par date_commande DESC. Créez les index appropriés et expliquez l'impact sur les performances.

Voir la solution 
-- Index composé pour les requêtes : SELECT ... WHERE client_id = X ORDER BY date_commande DESC
CREATE INDEX idx_commandes_client_date
ON commandes(client_id, date_commande DESC);

-- Cet index est efficace car :
-- 1. client_id filtre d'abord les commandes du client (B-tree)
-- 2. date_commande DESC permet le tri sans opération supplémentaire
-- Le moteur peut servir la requête uniquement via l'index (Index Only Scan)

-- Vérification avec EXPLAIN :
EXPLAIN ANALYZE
SELECT id, montant, date_commande
FROM commandes
WHERE client_id = 42
ORDER BY date_commande DESC
LIMIT 20;

-- Sans index : Seq Scan → parcours des 10M de lignes (~secondes)
-- Avec index : Index Scan → accès direct (~millisecondes)

-- Index sur le statut (cardinalité faible — moins efficace)
-- Utiliser un index partiel pour les commandes en cours uniquement :
CREATE INDEX idx_commandes_en_cours
ON commandes(client_id)
WHERE statut = 'en_cours';

Exercice 5 — MongoDB : CRUD complet

Implémentez un CRUD complet pour une collection produits avec MongoDB et Mongoose. Chaque produit a un nom, un prix, une catégorie et un tableau de tags. Ajoutez une méthode de recherche par tag et une agrégation pour le prix moyen par catégorie.

Voir la solution 
const mongoose = require('mongoose');

const produitSchema = new mongoose.Schema({
  nom:       { type: String, required: true },
  prix:      { type: Number, required: true, min: 0 },
  categorie: { type: String, required: true },
  tags:      [String],
  cree_le:   { type: Date, default: Date.now }
});

const Produit = mongoose.model('Produit', produitSchema);

// Créer
async function creer(data) {
  return Produit.create(data);
}

// Lire tous (avec pagination)
async function lister(page = 1, limite = 10) {
  return Produit.find()
    .skip((page - 1) * limite)
    .limit(limite)
    .sort({ cree_le: -1 });
}

// Rechercher par tag
async function parTag(tag) {
  return Produit.find({ tags: tag });
}

// Modifier
async function modifier(id, data) {
  return Produit.findByIdAndUpdate(id, data, { new: true });
}

// Supprimer
async function supprimer(id) {
  return Produit.findByIdAndDelete(id);
}

// Agrégation : prix moyen par catégorie
async function prixMoyenParCategorie() {
  return Produit.aggregate([
    { $group: {
      _id: '$categorie',
      prixMoyen: { $avg: '$prix' },
      count: { $sum: 1 }
    }},
    { $sort: { prixMoyen: -1 } }
  ]);
}
