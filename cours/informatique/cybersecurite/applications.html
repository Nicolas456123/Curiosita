01

Introduction

La sécurité dès la conception

La sécurité des applications (AppSec) vise à rendre les logiciels résistants aux attaques dès leur conception, tout au long de leur développement et jusqu'à leur déploiement en production. Plutôt que de corriger les failles après coup, l'approche moderne intègre la sécurité à chaque étape du cycle de vie du logiciel — c'est le principe du « shift left ».

Les applications web et les API sont aujourd'hui les cibles principales des cyberattaques. Elles exposent directement la logique métier, les données utilisateur et les systèmes internes. Une seule faille dans une API peut compromettre des millions de comptes. La sécurité applicative est donc un enjeu stratégique pour toute organisation.

Shift Left : intégrer la sécurité le plus tôt possible dans le cycle de développement. Corriger une vulnérabilité en phase de conception coûte 6 fois moins cher qu'en production. Le DevSecOps automatise les tests de sécurité dans la chaîne CI/CD pour détecter les failles avant le déploiement.

Ce cours couvre les pratiques essentielles : l'approche DevSecOps, la validation et la sanitisation des entrées, les mécanismes d'authentification et d'autorisation, la politique CORS, la sécurité des API REST et les en-têtes HTTP de sécurité.

Le cours

Sécuriser le développement logiciel

DevSecOps

Le DevSecOps intègre la sécurité dans chaque phase du cycle DevOps, au lieu de la traiter comme une étape séparée en fin de processus. L'objectif est d'automatiser les contrôles de sécurité pour qu'ils soient aussi naturels que les tests unitaires.

Phase de code : analyse statique du code (SAST — Static Application Security Testing) pour détecter les vulnérabilités dans le code source avant même l'exécution. Outils : SonarQube, Semgrep, Bandit (Python), ESLint Security (JavaScript).

Phase de build : scan des dépendances (SCA — Software Composition Analysis) pour identifier les bibliothèques avec des vulnérabilités connues. Outils : Snyk, OWASP Dependency-Check, npm audit, pip-audit.

Phase de test : analyse dynamique (DAST — Dynamic Application Security Testing) testant l'application en cours d'exécution. Outils : OWASP ZAP, Burp Suite, Nuclei.

Phase de déploiement : scan des images Docker pour les vulnérabilités, vérification des configurations (IaC security), politiques de déploiement. Outils : Trivy, Checkov, OPA (Open Policy Agent).

Pipeline CI/CD sécurisé (GitHub Actions)
# .github/workflows/security.yml
name: Security Pipeline
on: [push, pull_request]

jobs:
  sast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Analyse statique (Semgrep)
        run: semgrep scan --config auto --error

  dependency-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Scan des dépendances
        run: |
          pip install pip-audit
          pip-audit --strict       # Python
          npm audit --audit-level=high  # JavaScript

  container-scan:
    runs-on: ubuntu-latest
    steps:
      - name: Scan image Docker (Trivy)
        run: trivy image --severity HIGH,CRITICAL myapp:latest

  dast:
    runs-on: ubuntu-latest
    needs: [sast, dependency-check]
    steps:
      - name: Test dynamique (OWASP ZAP)
        run: zap-baseline.py -t https://staging.myapp.com

Validation et sanitisation des entrées

La règle d'or de la sécurité applicative : ne jamais faire confiance aux entrées utilisateur. Toute donnée provenant de l'extérieur (formulaires, URL, en-têtes, cookies, API) doit être validée et sanitisée avant utilisation.

Validation : vérifier que l'entrée correspond au format attendu (type, longueur, plage de valeurs, format regex). Rejeter tout ce qui ne correspond pas. Approche whitelist (n'accepter que ce qui est explicitement autorisé) plutôt que blacklist.

Sanitisation : nettoyer l'entrée en supprimant ou en échappant les caractères dangereux. L'échappement dépend du contexte de sortie (HTML, SQL, JavaScript, URL).

Python — Validation robuste
import re
from email_validator import validate_email, EmailNotValidError

def valider_inscription(donnees):
    erreurs = []

    # Validation du nom (lettres, espaces, tirets, 2-50 caractères)
    nom = donnees.get('nom', '').strip()
    if not re.match(r'^[a-zA-ZÀ-ÿ\s\-]{2,50}$', nom):
        erreurs.append("Nom invalide")

    # Validation de l'email (bibliothèque spécialisée)
    try:
        email = validate_email(donnees.get('email', '')).email
    except EmailNotValidError:
        erreurs.append("Email invalide")

    # Validation de l'âge (entier entre 13 et 120)
    try:
        age = int(donnees.get('age', 0))
        if not (13 <= age <= 120):
            erreurs.append("Âge hors limites")
    except ValueError:
        erreurs.append("Âge invalide")

    # Validation du mot de passe
    mdp = donnees.get('password', '')
    if len(mdp) < 12:
        erreurs.append("Mot de passe trop court (12 min)")
    if not re.search(r'[A-Z]', mdp):
        erreurs.append("Il faut au moins une majuscule")
    if not re.search(r'[0-9]', mdp):
        erreurs.append("Il faut au moins un chiffre")

    return erreurs  # Liste vide = validation OK

Authentification et autorisation

L'authentification vérifie l'identité (« qui êtes-vous ? ») tandis que l'autorisation vérifie les permissions (« avez-vous le droit ? »). Ce sont deux mécanismes distincts et complémentaires.

Authentification multifacteur (MFA) : combiner au moins deux facteurs parmi : ce que l'on sait (mot de passe), ce que l'on possède (téléphone, clé FIDO2), ce que l'on est (biométrie). Le MFA bloque la quasi-totalité des attaques par credential stuffing.

JWT (JSON Web Token) : format standardisé pour les tokens d'authentification. Un JWT contient trois parties encodées en Base64 : l'en-tête (algorithme), la charge utile (claims) et la signature. Attention aux erreurs classiques : ne pas vérifier la signature, accepter l'algorithme « none », stocker des données sensibles dans le payload.

OAuth 2.0 / OpenID Connect : protocoles de délégation d'autorisation et d'authentification. OAuth 2.0 permet à un utilisateur d'autoriser une application tierce à accéder à ses ressources sans partager son mot de passe. OpenID Connect ajoute une couche d'identité au-dessus d'OAuth.

JWT — Structure et vérification
# Structure d'un JWT (3 parties séparées par des points)

.

    "iss": "myapp.com"

CORS (Cross-Origin Resource Sharing)

La politique Same-Origin du navigateur interdit par défaut les requêtes JavaScript vers un domaine différent de celui de la page. CORS est un mécanisme qui permet de relâcher cette restriction de manière contrôlée.

Quand un site app.com fait une requête AJAX vers api.com, le navigateur envoie d'abord une requête « preflight » (OPTIONS) pour vérifier si le serveur autorise cette origine. Le serveur répond avec les en-têtes CORS appropriés.

CORS — Configuration sécurisée
# DANGEREUX : autoriser toutes les origines
Access-Control-Allow-Origin: *
# → N'importe quel site peut appeler votre API

# SÉCURISÉ : whitelist d'origines autorisées
Access-Control-Allow-Origin: https://app.mondomaine.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 3600

# Python Flask — Implémentation
from flask_cors import CORS

ORIGINES_AUTORISEES = [
    "https://app.mondomaine.com",
    "https://admin.mondomaine.com"
]

CORS(app, origins=ORIGINES_AUTORISEES,
     methods=["GET", "POST", "PUT", "DELETE"],
     allow_headers=["Content-Type", "Authorization"],
     supports_credentials=True)

# ERREURS COURANTES :
# - Allow-Origin: * avec credentials → bloqué par les navigateurs
# - Refléter l'en-tête Origin sans validation → bypass total
# - Oublier de vérifier les sous-domaines (evil.mondomaine.com)

Sécurité des API REST

Les API sont devenues la colonne vertébrale des applications modernes, mais elles exposent une surface d'attaque considérable. L'OWASP a publié un Top 10 spécifique aux API.

Authentification : chaque requête API doit être authentifiée. Les tokens Bearer (JWT) ou les clés API sont les mécanismes les plus courants. Ne jamais transmettre de credentials dans l'URL (visibles dans les logs).

Rate limiting : limiter le nombre de requêtes par utilisateur pour prévenir les abus (brute force, scraping, DDoS applicatif). Implémenter des limites par IP, par token et par endpoint.

Validation des entrées : valider strictement chaque paramètre selon un schéma (JSON Schema, OpenAPI). Rejeter les champs inconnus, vérifier les types, les tailles et les formats.

API sécurisée — Bonnes pratiques
# Exemple d'API REST sécurisée avec Flask
from flask import Flask, request, jsonify
from flask_limiter import Limiter
from functools import wraps

app = Flask(__name__)
limiter = Limiter(app, default_limits=["100/hour"])

# Middleware d'authentification
def auth_requise(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        token = request.headers.get('Authorization', '')
        if not token.startswith('Bearer '):
            return jsonify({"error": "Token manquant"}), 401
        # Vérifier et décoder le JWT...
        return f(*args, **kwargs)
    return decorated

En-têtes HTTP de sécurité

Les en-têtes HTTP de sécurité constituent une couche de défense supplémentaire côté navigateur. Ils permettent de limiter les attaques XSS, clickjacking, sniffing de type MIME et autres.

En-têtes de sécurité essentiels
# Content-Security-Policy — Prévention XSS
Content-Security-Policy: default-src 'self';
  script-src 'self' https://cdn.trusted.com;
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  connect-src 'self' https://api.mondomaine.com;
  frame-ancestors 'none'

# Strict-Transport-Security — Forcer HTTPS
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
# → Le navigateur refusera toute connexion HTTP pendant 1 an

# X-Content-Type-Options — Empêcher le sniffing MIME
X-Content-Type-Options: nosniff

# X-Frame-Options — Prévention du clickjacking
X-Frame-Options: DENY
# ou frame-ancestors 'none' dans CSP (plus moderne)

# Referrer-Policy — Contrôler les informations de referer
Referrer-Policy: strict-origin-when-cross-origin

# Permissions-Policy — Contrôler les fonctionnalités du navigateur
Permissions-Policy: camera=(), microphone=(), geolocation=(self)

# Vérifiez vos en-têtes : securityheaders.com

Gestion sécurisée des sessions et des secrets

Les sessions et les secrets (clés API, mots de passe de base de données) sont des cibles de choix pour les attaquants. Leur gestion correcte est cruciale.

Sessions : utiliser des identifiants de session aléatoires (128 bits minimum), régénérer l'ID après authentification (prévention de la fixation de session), configurer les cookies avec les attributs HttpOnly, Secure, SameSite=Strict et une expiration raisonnable.

Secrets : ne jamais stocker de secrets dans le code source ou les fichiers de configuration versionnés. Utiliser des gestionnaires de secrets (HashiCorp Vault, AWS Secrets Manager, variables d'environnement). Scanner régulièrement le dépôt avec des outils comme git-secrets ou truffleHog.

Checklist sécurité applicative :
Validation de toutes les entrées (whitelist) | Requêtes paramétrées (anti-injection) | Échappement des sorties (anti-XSS) | Authentification MFA | Autorisation par rôles (RBAC) | Cookies sécurisés (HttpOnly, Secure, SameSite) | En-têtes HTTP de sécurité (CSP, HSTS) | Rate limiting | Journalisation des événements de sécurité | Scan des dépendances en CI/CD

📊 Sécurité des applications — Quiz Score : 0 / 6

Question 1 / 6

Que signifie le principe « shift left » en DevSecOps ?

Le « shift left » consiste à déplacer la sécurité vers la gauche du cycle de développement (c'est-à-dire le plus tôt possible). Détecter une faille en phase de conception ou de code est beaucoup moins coûteux qu'en production.

Question 2 / 6

Quel outil analyse le code source sans l'exécuter pour détecter des vulnérabilités ?

Le SAST analyse le code source ou compilé sans l'exécuter, identifiant les vulnérabilités potentielles dès la phase de développement. Le DAST, à l'inverse, teste l'application en cours d'exécution.

Question 3 / 6

Pourquoi est-il dangereux de configurer CORS avec Access-Control-Allow-Origin: * ?

Avec Allow-Origin: *, tout site malveillant peut envoyer des requêtes à votre API depuis le navigateur de la victime. Si l'utilisateur est authentifié, l'attaquant pourrait accéder à ses données via CSRF ou des requêtes cross-origin.

Question 4 / 6

Quel attribut de cookie empêche JavaScript d'accéder au cookie de session ?

L'attribut HttpOnly empêche JavaScript d'accéder au cookie via document.cookie. Même en cas de XSS, l'attaquant ne peut pas voler le cookie de session. Secure force HTTPS, SameSite protège contre le CSRF.

Question 5 / 6

Quelle est la meilleure approche pour valider les entrées utilisateur ?

L'approche whitelist est plus sûre car elle n'accepte que ce qui est explicitement autorisé. La blacklist peut être contournée par des encodages alternatifs ou de nouveaux vecteurs d'attaque. La validation côté front-end est facilement contournable.

Question 6 / 6

Où faut-il stocker les secrets (clés API, mots de passe de bases de données) ?

Les secrets doivent être gérés par des outils dédiés (HashiCorp Vault, AWS Secrets Manager, etc.) ou au minimum dans des variables d'environnement. Jamais dans le code source ou les fichiers versionnés — les dépôts Git conservent tout l'historique.

Exercice 4 — En-têtes de sécurité

Votre site web est vulnérable au clickjacking, au XSS et ne force pas HTTPS. Écrivez le middleware (Python Flask ou Express.js) qui ajoute tous les en-têtes de sécurité recommandés à chaque réponse. Expliquez le rôle de chaque en-tête.

Voir la solution 
# Flask — Middleware d'en-têtes de sécurité

@app.after_request
def ajouter_entetes_securite(response):

    # 1. CSP — Bloque les scripts non autorisés (anti-XSS)
    response.headers['Content-Security-Policy'] = (
        "default-src 'self'; "
        "script-src 'self'; "
        "style-src 'self' 'unsafe-inline'; "
        "img-src 'self' data: https:; "
        "font-src 'self' https://fonts.gstatic.com; "
        "connect-src 'self' https://api.monsite.com; "
        "frame-ancestors 'none'; "   # Anti-clickjacking
        "base-uri 'self'; "
        "form-action 'self'"
    )

    # 2. HSTS — Force HTTPS pendant 1 an
    response.headers['Strict-Transport-Security'] = (
        'max-age=31536000; includeSubDomains; preload'
    )

    # 3. Empêche le sniffing de type MIME
    response.headers['X-Content-Type-Options'] = 'nosniff'

    # 4. Anti-clickjacking (fallback pour anciens navigateurs)
    response.headers['X-Frame-Options'] = 'DENY'

    # 5. Contrôle du Referer envoyé
    response.headers['Referrer-Policy'] = (
        'strict-origin-when-cross-origin'
    )

    # 6. Désactive les fonctionnalités non nécessaires
    response.headers['Permissions-Policy'] = (
        'camera=(), microphone=(), geolocation=(self), '
        'payment=(self)'
    )

    # 7. Protection XSS navigateur (legacy)
    response.headers['X-XSS-Protection'] = '0'
    # Note : désactivé car CSP est plus efficace
    # et X-XSS-Protection peut créer des vulnérabilités

    return response

# Vérification : https://securityheaders.com/?q=monsite.com
# Objectif : obtenir la note A+

        auth = request.headers.get('Authorization', '')
        if not auth.startswith('Bearer '):
            return jsonify({"error": "Non authentifié"}), 401
        try:
            payload = jwt.decode(
                auth[7:], SECRET, algorithms=["HS256"]
            )
            g.user_id = payload['user_id']
        except jwt.InvalidTokenError:
            return jsonify({"error": "Token invalide"}), 401
        return f(*args, **kwargs)
    return decorated

# Validation
def valider_tache(data):
    titre = data.get('titre', '').strip()
    if not titre or len(titre) > 200:
        return None, "Titre invalide (1-200 caractères)"
    # Sanitisation basique
    titre = re.sub(r'[<>]', '', titre)
    return {"titre": titre, "fait": bool(data.get('fait', False))}, None

@app.route('/api/taches', methods=['POST'])
@limiter.limit("30/minute")
@auth_requise
def creer():
    tache, erreur = valider_tache(request.json or {})
    if erreur:
        return jsonify({"error": erreur}), 400
    tache['user_id'] = g.user_id  # Associer au user courant
    result = db.creer_tache(tache)
    logger.info(f"POST /taches par user {g.user_id}")
    return jsonify(result), 201

@auth_requise

    tache = db.get_tache(id)
    if not tache or tache['user_id'] != g.user_id:
