Introduction

Réutiliser et spécialiser

L'héritage est un mécanisme fondamental de la POO qui permet de créer une nouvelle classe (classe enfant ou dérivée) à partir d'une classe existante (classe parente ou de base). La classe enfant hérite automatiquement de tous les attributs et méthodes de la classe parente, puis peut les compléter ou les redéfinir.

Ce mécanisme répond au principe DRY (Don't Repeat Yourself) : au lieu de dupliquer du code commun dans plusieurs classes, on le factorise dans une classe parente. Les classes enfants ne définissent que ce qui leur est spécifique.

Relation « est-un » : l'héritage modélise une relation de spécialisation. Un Chien est un Animal, un Carre est un Rectangle. Si la relation n'est pas naturelle, l'héritage n'est probablement pas le bon choix — préférez la composition (un objet contient un autre objet).

Le cours

Hiérarchies de classes en Python

Héritage simple

Pour hériter d'une classe, on place le nom de la classe parente entre parenthèses après le nom de la classe enfant :

class Animal:
    def __init__(self, nom, age):
        self.nom = nom
        self.age = age

    def se_presenter(self):
        return f"Je suis {self.nom}, j'ai {self.age} ans."

    def parler(self):
        return "..."

class Chien(Animal):
    def parler(self):
        return f"{self.nom} dit : Wouf !"

class Chat(Animal):
    def parler(self):
        return f"{self.nom} dit : Miaou !"

# Utilisation
rex = Chien("Rex", 5)
print(rex.se_presenter())  # Je suis Rex, j'ai 5 ans.  (hérité)
print(rex.parler())         # Rex dit : Wouf !          (redéfini)

felix = Chat("Félix", 3)
print(felix.parler())       # Félix dit : Miaou !

print(isinstance(rex, Chien))   # True
print(isinstance(rex, Animal))  # True (un Chien est un Animal)

La classe Chien hérite de se_presenter() sans la réécrire, mais redéfinit (override) la méthode parler().

La fonction super()

super() permet d'appeler une méthode de la classe parente depuis la classe enfant. C'est essentiel quand la classe enfant redéfinit __init__ mais veut conserver l'initialisation du parent :

class Vehicule:
    def __init__(self, marque, modele, annee):
        self.marque = marque
        self.modele = modele
        self.annee = annee

    def description(self):
        return f"{self.marque} {self.modele} ({self.annee})"

class Voiture(Vehicule):
    def __init__(self, marque, modele, annee, nb_portes):
        super().__init__(marque, modele, annee)  # appel au parent
        self.nb_portes = nb_portes               # attribut spécifique

    def description(self):
        base = super().description()  # réutilise la méthode parente
        return f"{base} — {self.nb_portes} portes"

class VoitureElectrique(Voiture):
    def __init__(self, marque, modele, annee, nb_portes, autonomie_km):
        super().__init__(marque, modele, annee, nb_portes)
        self.autonomie_km = autonomie_km

    def description(self):
        base = super().description()
        return f"{base} — autonomie {self.autonomie_km} km"

# Chaîne d'héritage sur 3 niveaux
tesla = VoitureElectrique("Tesla", "Model 3", 2023, 4, 580)
print(tesla.description())
# Tesla Model 3 (2023) — 4 portes — autonomie 580 km

Règle d'or : appelez toujours super().__init__(...) dans le __init__ de la classe enfant. Sinon, les attributs de la classe parente ne seront pas initialisés et vous obtiendrez des AttributeError.

Redéfinition de méthodes (override)

Une classe enfant peut redéfinir n'importe quelle méthode héritée. Python utilisera toujours la version la plus spécifique (celle de la classe de l'objet) :

class Forme:
    def aire(self):
        raise NotImplementedError("Les sous-classes doivent implémenter aire()")

class Rectangle(Forme):
    def __init__(self, largeur, hauteur):
        self.largeur = largeur
        self.hauteur = hauteur

    def aire(self):
        return self.largeur * self.hauteur

class Carre(Rectangle):
    def __init__(self, cote):
        super().__init__(cote, cote)  # un carré est un rectangle de côtés égaux

# Pas besoin de redéfinir aire() dans Carre — celle de Rectangle fonctionne
c = Carre(5)
print(c.aire())  # 25

Vérifier la hiérarchie

Python fournit des outils pour inspecter les relations d'héritage :

print(isinstance(tesla, VoitureElectrique))  # True
print(isinstance(tesla, Voiture))             # True
print(isinstance(tesla, Vehicule))            # True

print(issubclass(Voiture, Vehicule))          # True
print(issubclass(Vehicule, Voiture))          # False

# Voir la chaîne de classes parentes
print(VoitureElectrique.__mro__)
# (<class 'VoitureElectrique'>, <class 'Voiture'>,
#  <class 'Vehicule'>, <class 'object'>)

Toutes les classes Python héritent implicitement de object, la classe de base universelle.

Héritage multiple

Python permet à une classe d'hériter de plusieurs classes parentes simultanément :

class Nageur:
    def nager(self):
        return f"{self.nom} nage."

class Coureur:
    def courir(self):
        return f"{self.nom} court."

class Triathlete(Nageur, Coureur):
    def __init__(self, nom):
        self.nom = nom

    def compete(self):
        return f"{self.nager()} {self.courir()}"

t = Triathlete("Alice")
print(t.compete())  # Alice nage. Alice court.

Le MRO (Method Resolution Order)

Quand plusieurs classes parentes définissent la même méthode, Python utilise l'algorithme C3 Linearization pour déterminer l'ordre de recherche. C'est le MRO (Method Resolution Order) :

class A:
    def qui(self):
        return "A"

class B(A):
    def qui(self):
        return "B"

class C(A):
    def qui(self):
        return "C"

class D(B, C):
    pass  # ne redéfinit pas qui()

d = D()
print(d.qui())  # B — trouvé dans B avant C

# L'ordre de résolution complet :
print(D.__mro__)
# (<class 'D'>, <class 'B'>, <class 'C'>, <class 'A'>, <class 'object'>)

# Python cherche dans l'ordre : D → B → C → A → object
# Il trouve qui() dans B et s'arrête

Le problème du diamant : quand D hérite de B et C, qui héritent toutes deux de A, le MRO garantit que A n'est visité qu'une seule fois et dans le bon ordre. C'est pour cela que super() est préférable aux appels directs au parent.

Héritage vs composition

L'héritage n'est pas toujours la meilleure solution. La composition (un objet possède un autre objet) est souvent préférable :

# Mauvais usage de l'héritage (une voiture n'est pas un moteur)
class Moteur:
    def demarrer(self):
        return "Moteur démarré"

# ❌ Héritage inapproprié
class MauvaiseVoiture(Moteur):
    pass

# ✓ Composition : une voiture A UN moteur
class Moteur:
    def __init__(self, puissance):
        self.puissance = puissance

    def demarrer(self):
        return f"Moteur {self.puissance}ch démarré"

class BonneVoiture:
    def __init__(self, marque, puissance):
        self.marque = marque
        self.moteur = Moteur(puissance)  # composition

    def demarrer(self):
        return f"{self.marque} : {self.moteur.demarrer()}"

v = BonneVoiture("Peugeot", 130)
print(v.demarrer())  # Peugeot : Moteur 130ch démarré

Règle pratique : utilisez l'héritage pour les relations « est-un » et la composition pour les relations « a-un » ou « utilise-un ».

🐍 Héritage — Quiz Score : 0 / 6

Question 1 / 6

Comment une classe enfant hérite-t-elle d'une classe parente en Python ?

En Python, on indique l'héritage en plaçant la classe parente entre parenthèses : class Enfant(Parent). Le mot-clé extends est utilisé en Java, pas en Python.

Question 2 / 6

À quoi sert la fonction super() ?

super() retourne un objet proxy qui permet d'appeler les méthodes de la classe parente. C'est indispensable pour réutiliser l'initialisation du parent dans __init__ de l'enfant.

Question 3 / 6

Que signifie MRO en Python ?

Le MRO (Method Resolution Order) détermine l'ordre dans lequel Python cherche une méthode dans la hiérarchie de classes, surtout en cas d'héritage multiple. Il utilise l'algorithme C3 Linearization.

Question 4 / 6

Si class D(B, C) et que B et C définissent toutes deux une méthode action(), laquelle sera appelée par D().action() ?

Le MRO suit l'ordre de déclaration des parents : D → B → C. Python trouve action() dans B en premier et l'utilise. C'est pourquoi l'ordre des parents dans la déclaration est important.

Question 5 / 6

Quelle fonction permet de vérifier si un objet est une instance d'une classe ou de ses classes parentes ?

isinstance(obj, Classe) retourne True si obj est une instance de Classe ou d'une de ses sous-classes. issubclass() vérifie la relation entre classes, pas entre un objet et une classe.

Question 6 / 6

Quand devrait-on préférer la composition à l'héritage ?

La composition modélise les relations « a-un » (une voiture a un moteur). L'héritage modélise les relations « est-un » (un chien est un animal). Utiliser l'héritage pour une relation « a-un » est un anti-pattern courant.

Exercice 1 — Hiérarchie d'animaux

Créez une classe de base Animal avec nom et age. Puis créez trois classes enfants : Chien, Chat et Oiseau. Chaque sous-classe redéfinit parler() et possède un attribut spécifique (race pour Chien, interieur pour Chat, envergure pour Oiseau).

Voir la solution 
class Animal:
    def __init__(self, nom, age):
        self.nom = nom
        self.age = age

    def parler(self):
        return "..."

class Chien(Animal):
    def __init__(self, nom, age, race):
        super().__init__(nom, age)
        self.race = race

    def parler(self):
        return f"{self.nom} fait : Wouf !"

class Chat(Animal):
    def __init__(self, nom, age, interieur=True):
        super().__init__(nom, age)
        self.interieur = interieur

    def parler(self):
        return f"{self.nom} fait : Miaou !"

class Oiseau(Animal):
    def __init__(self, nom, age, envergure):
        super().__init__(nom, age)
        self.envergure = envergure

    def parler(self):
        return f"{self.nom} fait : Cui cui !"

# Test
animaux = [Chien("Rex", 5, "Labrador"), Chat("Mimi", 3), Oiseau("Piou", 1, 30)]
for a in animaux:
    print(a.parler())

Exercice 2 — Employés et salaires

Créez une classe Employe avec nom et salaire_base. Créez deux sous-classes : Manager (qui a un bonus en pourcentage) et Developpeur (qui a une prime fixe par projet). Chaque sous-classe redéfinit salaire_total().

Voir la solution 
class Employe:
    def __init__(self, nom, salaire_base):
        self.nom = nom
        self.salaire_base = salaire_base

    def salaire_total(self):
        return self.salaire_base

class Manager(Employe):
    def __init__(self, nom, salaire_base, bonus_pct):
        super().__init__(nom, salaire_base)
        self.bonus_pct = bonus_pct  # ex: 0.20 pour 20%

    def salaire_total(self):
        return self.salaire_base * (1 + self.bonus_pct)

class Developpeur(Employe):
    def __init__(self, nom, salaire_base, prime_projet, nb_projets):
        super().__init__(nom, salaire_base)
        self.prime_projet = prime_projet
        self.nb_projets = nb_projets

    def salaire_total(self):
        return self.salaire_base + self.prime_projet * self.nb_projets

# Test
m = Manager("Alice", 5000, 0.20)
d = Developpeur("Bob", 4000, 500, 3)
print(f"{m.nom} : {m.salaire_total()}€")  # Alice : 6000.0€
print(f"{d.nom} : {d.salaire_total()}€")  # Bob : 5500€

Exercice 3 — Héritage multiple — Mixin de log

Créez un mixin LogMixin avec une méthode log(message) qui affiche un message avec un timestamp. Créez une classe Serveur(LogMixin) qui utilise ce mixin pour logger les connexions et déconnexions.

Voir la solution 
from datetime import datetime

class LogMixin:
    def log(self, message):
        horodatage = datetime.now().strftime("%H:%M:%S")
        print(f"[{horodatage}] {self.__class__.__name__}: {message}")

class Serveur(LogMixin):
    def __init__(self, nom):
        self.nom = nom
        self.connectes = []

    def connecter(self, utilisateur):
        self.connectes.append(utilisateur)
        self.log(f"{utilisateur} s'est connecté")

    def deconnecter(self, utilisateur):
        self.connectes.remove(utilisateur)
        self.log(f"{utilisateur} s'est déconnecté")

# Test
s = Serveur("MonServeur")
s.connecter("Alice")   # [14:30:25] Serveur: Alice s'est connecté
s.connecter("Bob")     # [14:30:25] Serveur: Bob s'est connecté
s.deconnecter("Alice") # [14:30:25] Serveur: Alice s'est déconnecté

Exercice 4 — Chaîne d'héritage et super()

Créez une hiérarchie Figure → Polygone → PolygoneRegulier. Figure a une couleur. Polygone ajoute un nombre de côtés. PolygoneRegulier ajoute la longueur d'un côté et calcule le périmètre. Utilisez super() à chaque niveau.

Voir la solution 
class Figure:
    def __init__(self, couleur):
        self.couleur = couleur

    def description(self):
        return f"Figure {self.couleur}"

class Polygone(Figure):
    def __init__(self, couleur, nb_cotes):
        super().__init__(couleur)
        self.nb_cotes = nb_cotes

    def description(self):
        return f"Polygone {self.couleur} à {self.nb_cotes} côtés"

class PolygoneRegulier(Polygone):
    def __init__(self, couleur, nb_cotes, longueur_cote):
        super().__init__(couleur, nb_cotes)
        self.longueur_cote = longueur_cote

    def perimetre(self):
        return self.nb_cotes * self.longueur_cote

    def description(self):
        base = super().description()
        return f"{base} (côté={self.longueur_cote}, périmètre={self.perimetre()})"

# Test
hexagone = PolygoneRegulier("bleu", 6, 4)
print(hexagone.description())
# Polygone bleu à 6 côtés (côté=4, périmètre=24)

Exercice 5 — Explorer le MRO

Créez la hiérarchie suivante : A définit action() retournant "A". B(A) et C(A) redéfinissent action(). D(B, C) ne la redéfinit pas. Affichez le MRO de D et prédisez le résultat de D().action(). Ensuite, créez E(C, B) et comparez.

Voir la solution 
class A:
    def action(self):
        return "A"

class B(A):
    def action(self):
        return "B"

class C(A):
    def action(self):
        return "C"

class D(B, C):
    pass

class E(C, B):
    pass

# Test
print(D().action())  # B (premier parent)
print(E().action())  # C (premier parent)

print("MRO de D:", [cls.__name__ for cls in D.__mro__])
# MRO de D: ['D', 'B', 'C', 'A', 'object']

print("MRO de E:", [cls.__name__ for cls in E.__mro__])
# MRO de E: ['E', 'C', 'B', 'A', 'object']

# L'ordre des parents dans la déclaration détermine la priorité
