Introduction

Le côté invisible du web

Le back-end (ou « côté serveur ») désigne tout ce qui se passe en coulisses d'une application web : réception des requêtes, traitement de la logique métier, accès aux bases de données et renvoi des réponses. Si le front-end est la vitrine, le back-end est l'atelier.

Architecture client-serveur : le navigateur (client) envoie des requêtes HTTP au serveur, qui les traite et renvoie des réponses. Le back-end gère la logique, les données, la sécurité et les performances. Les langages courants : JavaScript (Node.js), Python (Django, Flask), PHP (Laravel), Java (Spring), Ruby (Rails), Go.

Avec l'essor des API REST et des architectures découplées, le back-end moderne se concentre sur la création de services consommables par n'importe quel client (web, mobile, IoT), renforçant la réutilisabilité et la scalabilité.

Le cours

Serveurs, API et sécurité

Le protocole HTTP

HTTP (HyperText Transfer Protocol) est le protocole de communication du web. Chaque échange se compose d'une requête (méthode, URL, en-têtes, corps) et d'une réponse (code de statut, en-têtes, corps). Les méthodes principales sont :

GET — récupérer une ressource (lecture)
POST — créer une ressource
PUT — remplacer intégralement une ressource
PATCH — modifier partiellement une ressource
DELETE — supprimer une ressource

Codes de statut : 200 (OK), 201 (Created), 400 (Bad Request), 401 (Unauthorized), 404 (Not Found), 500 (Internal Server Error)

Node.js et Express

Node.js exécute JavaScript côté serveur grâce au moteur V8 de Chrome. Express est le framework minimaliste le plus populaire pour Node.js. Il gère le routage, les middlewares et les réponses HTTP.

JavaScript (Node.js + Express)
const express = require('express');
const app = express();

// Middleware pour parser le JSON
app.use(express.json());

// Route GET
app.get('/api/users', (req, res) => {
  const users = [
    { id: 1, nom: 'Alice' },
    { id: 2, nom: 'Bob' }
  ];
  res.json(users);
});

// Route POST
app.post('/api/users', (req, res) => {
  const { nom, email } = req.body;
  // Validation et sauvegarde en base...
  res.status(201).json({ id: 3, nom, email });
});

app.listen(3000, () => {
  console.log('Serveur démarré sur le port 3000');
});

API REST : principes et conception

Une API REST (Representational State Transfer) organise les ressources via des URLs et utilise les méthodes HTTP pour les opérations CRUD. Les principes REST incluent : des URLs prévisibles, l'absence d'état côté serveur (stateless), et des réponses au format JSON.

API REST — Convention des routes
GET    /api/articles        → Liste des articles
GET    /api/articles/:id    → Un article par ID
POST   /api/articles        → Créer un article
PUT    /api/articles/:id    → Remplacer un article
PATCH  /api/articles/:id    → Modifier un article
DELETE /api/articles/:id    → Supprimer un article

// Relations imbriquées
GET    /api/articles/:id/commentaires → Commentaires d'un article

// Filtrage, pagination, tri via query params
GET    /api/articles?categorie=tech&page=2&limit=10&sort=-date

Middlewares

Les middlewares sont des fonctions qui interceptent les requêtes avant qu'elles n'atteignent les routes. Ils servent à la journalisation, l'authentification, la validation, le CORS, la gestion des erreurs, etc.

JavaScript (Express)
// Middleware de journalisation
app.use((req, res, next) => {
  console.log(`${req.method} ${req.url}`);
  next(); // passer au middleware suivant
});

// Middleware d'authentification
function authMiddleware(req, res, next) {
  const token = req.headers.authorization;
  if (!token) {
    return res.status(401).json({ error: 'Token manquant' });
  }
  try {
    req.user = verifierToken(token);
    next();
  } catch (err) {
    res.status(403).json({ error: 'Token invalide' });
  }
}

// Appliquer sur les routes protégées
app.get('/api/profil', authMiddleware, (req, res) => {
  res.json(req.user);
});

Authentification et JWT

L'authentification vérifie l'identité de l'utilisateur. Les JSON Web Tokens (JWT) sont une méthode courante : après connexion, le serveur génère un token signé que le client envoie à chaque requête.

JavaScript
const jwt = require('jsonwebtoken');
const SECRET = process.env.JWT_SECRET;

// Connexion : générer un token
app.post('/api/login', async (req, res) => {
  const { email, motDePasse } = req.body;
  const user = await trouverUtilisateur(email);

  if (!user || !await verifierMdp(motDePasse, user.hash)) {
    return res.status(401).json({ error: 'Identifiants invalides' });
  }

  const token = jwt.sign(
    { id: user.id, role: user.role },
    SECRET,
    { expiresIn: '24h' }
  );
  res.json({ token });
});

// Vérifier un token
function verifierToken(token) {
  return jwt.verify(token.replace('Bearer ', ''), SECRET);
}

Sécurité des applications web

La sécurité est fondamentale côté serveur. Les attaques les plus courantes sont les injections SQL, le XSS (Cross-Site Scripting), le CSRF (Cross-Site Request Forgery) et les fuites de données.

Bonnes pratiques :
Validation des entrées — ne jamais faire confiance aux données client
Requêtes paramétrées — éviter les injections SQL
Hachage des mots de passe — utiliser bcrypt avec un sel (salt)
HTTPS — chiffrer toutes les communications
CORS — restreindre les origines autorisées
Rate limiting — limiter le nombre de requêtes par IP
Helmet — en-têtes HTTP de sécurité (Express)

Architecture et déploiement

Les applications modernes suivent souvent une architecture en microservices (services indépendants communiquant via API) ou monolithique (une seule application). Le déploiement utilise des conteneurs (Docker), des orchestrateurs (Kubernetes) ou des plateformes PaaS (Heroku, Railway, Vercel).

Variables d'environnement : les secrets (clés API, mots de passe de BDD) ne sont jamais codés en dur dans le code source. On utilise des fichiers .env et des bibliothèques comme dotenv pour les charger en mémoire.

Structure d'un projet Express :
/routes — définition des routes
/controllers — logique métier
/models — modèles de données (ORM)
/middlewares — fonctions intermédiaires
/config — configuration (BDD, auth)

Question 1 / 6

Quelle méthode HTTP est utilisée pour créer une nouvelle ressource ?

POST est la méthode standard pour créer une ressource. GET sert à lire, PUT à remplacer, PATCH à modifier partiellement, et DELETE à supprimer.

Question 2 / 6

Que signifie le code de statut HTTP 404 ?

404 Not Found signifie que le serveur n'a pas trouvé la ressource demandée. 400 = mauvaise requête, 401 = non autorisé, 500 = erreur interne du serveur.

Question 3 / 6

Qu'est-ce qu'un middleware dans Express ?

Un middleware est une fonction avec accès à req, res et next(). Il peut modifier la requête/réponse, terminer le cycle ou passer au middleware suivant. Exemples : authentification, journalisation, CORS.

Question 4 / 6

Pourquoi ne faut-il jamais stocker les mots de passe en clair dans la base de données ?

Si la base de données est compromise, les mots de passe en clair sont directement exploitables. On utilise des fonctions de hachage (bcrypt, Argon2) avec un sel unique pour stocker un hash irréversible.

Question 5 / 6

Qu'est-ce qu'un JWT (JSON Web Token) ?

Un JWT est composé de trois parties (header, payload, signature) encodées en Base64. Le serveur le signe avec une clé secrète et le vérifie à chaque requête, sans avoir à stocker l'état de session.

Question 6 / 6

Qu'est-ce que le principe « stateless » en API REST ?

« Stateless » signifie que le serveur ne conserve pas d'état entre les requêtes. Chaque requête est indépendante et doit contenir toute l'information nécessaire (token, paramètres). Cela facilite la scalabilité horizontale.

Exercice 2 — CRUD complet

Créez une API REST pour gérer une liste de tâches (todos) en mémoire. Implémentez les 5 routes : lister, lire une tâche, créer, modifier et supprimer. Chaque tâche a un id, un titre et un statut (fait/pas fait).

Voir la solution 
const express = require('express');
const app = express();
app.use(express.json());

let todos = [];
let nextId = 1;

// Lister toutes les tâches
app.get('/api/todos', (req, res) => {
  res.json(todos);
});

// Lire une tâche
app.get('/api/todos/:id', (req, res) => {
  const todo = todos.find(t => t.id === parseInt(req.params.id));
  if (!todo) return res.status(404).json({ error: 'Tâche non trouvée' });
  res.json(todo);
});

// Créer une tâche
app.post('/api/todos', (req, res) => {
  const { titre } = req.body;
  if (!titre) return res.status(400).json({ error: 'Titre requis' });
  const todo = { id: nextId++, titre, fait: false };
  todos.push(todo);
  res.status(201).json(todo);
});

// Modifier une tâche
app.patch('/api/todos/:id', (req, res) => {
  const todo = todos.find(t => t.id === parseInt(req.params.id));
  if (!todo) return res.status(404).json({ error: 'Tâche non trouvée' });
  if (req.body.titre !== undefined) todo.titre = req.body.titre;
  if (req.body.fait !== undefined) todo.fait = req.body.fait;
  res.json(todo);
});

// Supprimer une tâche
app.delete('/api/todos/:id', (req, res) => {
  const index = todos.findIndex(t => t.id === parseInt(req.params.id));
  if (index === -1) return res.status(404).json({ error: 'Tâche non trouvée' });
  todos.splice(index, 1);
  res.status(204).send();
});

app.listen(3000);

Exercice 3 — Middleware de validation

Créez un middleware qui valide le corps des requêtes POST vers /api/users. Le middleware doit vérifier que nom (3 caractères minimum), email (contient @) et age (nombre entre 0 et 150) sont présents et valides.

Voir la solution 
function validerUser(req, res, next) {
  const { nom, email, age } = req.body;
  const erreurs = [];

  if (!nom || nom.length < 3) {
    erreurs.push('Le nom doit avoir au moins 3 caractères');
  }
  if (!email || !email.includes('@')) {
    erreurs.push('Email invalide');
  }
  if (age === undefined || typeof age !== 'number' || age < 0 || age > 150) {
    erreurs.push('Âge invalide (0-150)');
  }

  if (erreurs.length > 0) {
    return res.status(400).json({ erreurs });
  }
  next();
}

app.post('/api/users', validerUser, (req, res) => {
  // À ce stade, les données sont validées
  res.status(201).json({ message: 'Utilisateur créé', data: req.body });
});

Exercice 4 — Authentification par token

Implémentez un système de login/inscription simplifié avec JWT. Route POST /api/register pour créer un utilisateur (hashage du mot de passe avec bcrypt), route POST /api/login pour se connecter et recevoir un token, et une route protégée GET /api/profil.

Voir la solution 
const bcrypt = require('bcrypt');
const jwt = require('jsonwebtoken');
const SECRET = 'ma_cle_secrete'; // En production : variable d'environnement

const users = [];

app.post('/api/register', async (req, res) => {
  const { nom, email, motDePasse } = req.body;
  if (users.find(u => u.email === email)) {
    return res.status(409).json({ error: 'Email déjà utilisé' });
  }
  const hash = await bcrypt.hash(motDePasse, 10);
  const user = { id: users.length + 1, nom, email, hash };
  users.push(user);
  res.status(201).json({ message: 'Compte créé' });
});

app.post('/api/login', async (req, res) => {
  const { email, motDePasse } = req.body;
  const user = users.find(u => u.email === email);
  if (!user || !await bcrypt.compare(motDePasse, user.hash)) {
    return res.status(401).json({ error: 'Identifiants invalides' });
  }
  const token = jwt.sign({ id: user.id }, SECRET, { expiresIn: '24h' });
  res.json({ token });
});

function auth(req, res, next) {
  const token = req.headers.authorization?.replace('Bearer ', '');
  if (!token) return res.status(401).json({ error: 'Token manquant' });
  try {
    req.user = jwt.verify(token, SECRET);
    next();
  } catch { res.status(403).json({ error: 'Token invalide' }); }
}

app.get('/api/profil', auth, (req, res) => {
  const user = users.find(u => u.id === req.user.id);
  res.json({ id: user.id, nom: user.nom, email: user.email });
});

Exercice 5 — Gestion des erreurs

Créez un middleware de gestion d'erreurs centralisé pour Express. Il doit attraper toutes les erreurs, les journaliser, et retourner une réponse JSON appropriée avec le bon code de statut. Créez aussi une classe AppError pour les erreurs personnalisées.

Voir la solution 
class AppError extends Error {
  constructor(message, statusCode) {
    super(message);
    this.statusCode = statusCode;
    this.isOperational = true;
  }
}

// Middleware de gestion d'erreurs (4 paramètres)
function errorHandler(err, req, res, next) {
  console.error(`[ERREUR] ${err.message}`);

  if (err.isOperational) {
    return res.status(err.statusCode).json({
      error: err.message
    });
  }

  // Erreur inattendue
  res.status(500).json({
    error: 'Erreur interne du serveur'
  });
}

// Utilisation dans les routes
app.get('/api/articles/:id', async (req, res, next) => {
  try {
    const article = await trouverArticle(req.params.id);
    if (!article) {
      throw new AppError('Article non trouvé', 404);
    }
    res.json(article);
  } catch (err) {
    next(err); // passe l'erreur au handler
  }
});

// Enregistrer en dernier
app.use(errorHandler);
