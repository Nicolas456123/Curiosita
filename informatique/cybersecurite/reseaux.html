Introduction

Le réseau comme champ de bataille

Le réseau est le système nerveux de toute infrastructure informatique. Chaque donnée échangée entre un utilisateur et un serveur, chaque email envoyé, chaque appel vidéo transite par des réseaux interconnectés. Sécuriser ces communications est un enjeu majeur car le réseau est par nature un espace partagé où les données peuvent être interceptées, modifiées ou bloquées.

Les attaques réseau sont parmi les plus anciennes et les plus fréquentes en cybersécurité. Du simple sniffing de paquets aux attaques DDoS sophistiquées, en passant par l'usurpation d'identité réseau (spoofing) et l'interception de communications (man-in-the-middle), les vecteurs d'attaque sont nombreux et en constante évolution.

Modèle de défense : la sécurité réseau repose sur une combinaison de prévention (pare-feu, segmentation), de détection (IDS, SIEM) et de réaction (plans d'incident, forensics réseau). Aucune mesure seule n'est suffisante.

Ce cours couvre les mécanismes de protection réseau essentiels : pare-feu, VPN, systèmes de détection et de prévention d'intrusion, protocoles sécurisés, et les principales attaques réseau avec leurs contre-mesures.

Le cours

Défenses et attaques réseau

Pare-feu (Firewall)

Un pare-feu est un dispositif (matériel ou logiciel) qui filtre le trafic réseau en fonction de règles prédéfinies. Il constitue la première ligne de défense entre un réseau de confiance (interne) et un réseau non fiable (Internet).

Pare-feu à filtrage de paquets : examine chaque paquet individuellement (IP source/destination, port, protocole). Rapide mais incapable d'analyser le contexte d'une connexion. Exemple : iptables sous Linux.

Pare-feu à état (stateful) : suit l'état des connexions TCP et ne laisse passer que les paquets appartenant à des connexions légitimes. Bloque les paquets « orphelins » ou incohérents.

Pare-feu applicatif (WAF) : opère au niveau de la couche applicative (HTTP, DNS). Capable d'inspecter le contenu des requêtes pour détecter des injections SQL, du XSS ou d'autres attaques applicatives.

iptables (Linux)
# Politique par défaut : tout bloquer
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# Autoriser le trafic loopback
iptables -A INPUT -i lo -j ACCEPT

# Autoriser les connexions établies et liées
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Autoriser SSH uniquement depuis le réseau interne
iptables -A INPUT -s 192.168.1.0/24 -p tcp --dport 22 -j ACCEPT

# Autoriser HTTP/HTTPS depuis n'importe où
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Journaliser les paquets rejetés
iptables -A INPUT -j LOG --log-prefix "[FW-DROP] "
iptables -A INPUT -j DROP

VPN (Virtual Private Network)

Un VPN crée un tunnel chiffré entre deux points du réseau, permettant de transmettre des données de manière confidentielle sur un réseau non sûr comme Internet. Le trafic est encapsulé et chiffré, rendant son interception inutile.

IPsec : protocole standard opérant au niveau de la couche réseau (couche 3). Deux modes : transport (chiffre uniquement la charge utile) et tunnel (chiffre tout le paquet, y compris les en-têtes). Utilisé pour les VPN site-à-site entre entreprises.

OpenVPN : solution open source basée sur TLS. Flexible, fonctionne sur UDP ou TCP, traverse facilement les pare-feu. Très populaire pour les VPN d'accès distant.

WireGuard : protocole moderne, minimaliste (environ 4000 lignes de code contre 100 000 pour OpenVPN). Performances supérieures et surface d'attaque réduite. Intégré au noyau Linux depuis la version 5.6.

WireGuard — Configuration
# /etc/wireguard/wg0.conf (Serveur)
[Interface]
Address = 10.0.0.1/24
ListenPort = 51820
PrivateKey = cle_privee_serveur_base64

# Activer le forwarding IP
PostUp = iptables -A FORWARD -i wg0 -j ACCEPT
PostDown = iptables -D FORWARD -i wg0 -j ACCEPT

[Peer]
PublicKey = cle_publique_client_base64
AllowedIPs = 10.0.0.2/32

# /etc/wireguard/wg0.conf (Client)
[Interface]
Address = 10.0.0.2/24
PrivateKey = cle_privee_client_base64

[Peer]
PublicKey = cle_publique_serveur_base64
Endpoint = 203.0.113.1:51820
AllowedIPs = 0.0.0.0/0  # Tout le trafic passe par le VPN

IDS et IPS

Un IDS (Intrusion Detection System) surveille le trafic réseau pour détecter des activités suspectes et génère des alertes. Un IPS (Intrusion Prevention System) va plus loin en bloquant automatiquement le trafic malveillant détecté.

Détection par signatures : compare le trafic à une base de données de motifs d'attaques connus (similaire à un antivirus). Très efficace contre les attaques connues, mais inefficace contre les attaques inédites (zero-day).

Détection par anomalies : établit un profil du trafic normal et alerte quand le comportement dévie significativement. Peut détecter des attaques inconnues, mais génère plus de faux positifs.

Snort et Suricata sont les IDS/IPS open source les plus utilisés. Suricata gère le multi-threading natif et l'inspection approfondie des protocoles.

Snort — Règle de détection
# Détecter une tentative de scan de port (SYN sans ACK)
alert tcp any any -> $HOME_NET any (
    msg:"Scan SYN détecté";
    flags:S;
    threshold: type threshold, track by_src, count 20, seconds 5;
    sid:1000001; rev:1;
)

# Détecter une tentative d'injection SQL basique
alert tcp any any -> $HOME_NET $HTTP_PORTS (
    msg:"Tentative injection SQL";
    content:"UNION SELECT"; nocase;
    content:"FROM"; nocase; distance:0;
    sid:1000002; rev:1;
)

# Détecter un transfert de zone DNS non autorisé
alert tcp any any -> $DNS_SERVERS 53 (
    msg:"Transfert de zone DNS (AXFR)";
    content:|00 FC|;
    sid:1000003; rev:1;
)

Segmentation réseau

La segmentation divise le réseau en zones isolées pour limiter la propagation d'une attaque. Si un poste de travail est compromis, l'attaquant ne doit pas pouvoir accéder directement aux serveurs de base de données.

VLAN (Virtual LAN) : sépare logiquement un réseau physique en plusieurs sous-réseaux. Chaque VLAN est un domaine de diffusion distinct. Le trafic entre VLAN passe par un routeur/pare-feu.

DMZ (Zone démilitarisée) : sous-réseau isolé entre Internet et le réseau interne, hébergeant les serveurs accessibles depuis l'extérieur (web, mail, DNS). Deux pare-feu protègent la DMZ : un face à Internet, un face au réseau interne.

Micro-segmentation : approche Zero Trust où chaque charge de travail (workload) a ses propres règles de sécurité. Le mouvement latéral d'un attaquant est rendu extrêmement difficile car chaque communication entre services est contrôlée.

Protocoles sécurisés

Remplacer les protocoles en clair par leurs équivalents sécurisés est une mesure fondamentale :

Protocoles : non sécurisé → sécurisé
Protocole          Non sécurisé    →    Sécurisé
─────────────────────────────────────────────────
Web                HTTP (80)       →    HTTPS (443)     TLS
Transfert fichier  FTP (21)        →    SFTP (22)       SSH
Shell distant      Telnet (23)     →    SSH (22)        Chiffré
Email envoi        SMTP (25)       →    SMTPS (465)     TLS
Email réception    POP3 (110)      →    POP3S (995)     TLS
                   IMAP (143)      →    IMAPS (993)     TLS
DNS                DNS (53)        →    DoH/DoT (443)   TLS
Réseau local       HTTP            →    802.1X          EAP

Règle d'or : si un protocole transmet en clair, il existe
une alternative chiffrée. Toujours préférer la version sécurisée.

Attaques réseau courantes

Les attaques réseau exploitent les faiblesses des protocoles et des configurations. Connaître ces attaques est essentiel pour s'en défendre.

Man-in-the-Middle (MITM) : l'attaquant s'interpose entre deux parties communicantes, interceptant et potentiellement modifiant les échanges. L'ARP spoofing sur un réseau local est la technique classique : l'attaquant associe son adresse MAC à l'adresse IP de la passerelle, redirigeant tout le trafic vers lui.

DDoS (Distributed Denial of Service) : saturer un service avec un volume massif de requêtes provenant de milliers de machines compromises (botnet). Les attaques volumétriques peuvent atteindre plusieurs térabits par seconde. Contre-mesures : CDN, filtrage en amont, services anti-DDoS spécialisés.

DNS poisoning : corrompre le cache DNS pour rediriger les utilisateurs vers des serveurs malveillants. L'utilisateur tape une URL légitime mais est redirigé vers un site de phishing. DNSSEC (signatures DNS) est la contre-mesure principale.

Scan de ports : l'attaquant sonde les ports d'un serveur pour identifier les services actifs et leurs versions. Nmap est l'outil de référence. Contre-mesure : fermer les ports inutiles, masquer les bannières de version, utiliser un IDS pour détecter les scans.

Architecture réseau sécurisée

Une architecture réseau bien conçue intègre la sécurité à chaque niveau :

Principes d'architecture sécurisée :
Segmentation : isoler les zones (production, développement, DMZ, management)
Redondance : éliminer les points de défaillance uniques (SPOF)
Surveillance : centraliser les logs (SIEM), monitorer le trafic en temps réel
Accès : authentification 802.1X sur les ports physiques, NAC (Network Access Control)
Chiffrement : TLS pour tout le trafic, VPN pour les accès distants

Exercice 1 — Règles de pare-feu

Un serveur web héberge un site HTTPS et un service SSH pour l'administration. Écrivez les règles iptables nécessaires en suivant le principe du moindre privilège : bloquer tout par défaut, n'autoriser que le strict nécessaire. Le SSH ne doit être accessible que depuis l'adresse IP de l'administrateur (192.168.1.100).

Voir la solution 
# Politique par défaut : tout bloquer
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# Autoriser le loopback (communication locale)
iptables -A INPUT -i lo -j ACCEPT

# Autoriser les connexions déjà établies
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Autoriser HTTPS depuis n'importe où (port 443)
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Autoriser SSH uniquement depuis l'IP admin
iptables -A INPUT -s 192.168.1.100 -p tcp --dport 22 -j ACCEPT

# Autoriser ICMP ping (optionnel, pour le monitoring)
iptables -A INPUT -p icmp --icmp-type echo-request -j ACCEPT

# Journaliser puis rejeter tout le reste
iptables -A INPUT -j LOG --log-prefix "[FW-DROP] "
iptables -A INPUT -j DROP

# Notes : pas de règle HTTP (port 80) car tout passe par HTTPS
# Redirection 80→443 gérée au niveau du serveur web (nginx)

Exercice 2 — Analyse de capture réseau

Avec Wireshark ou tcpdump, capturez le trafic lors d'une connexion HTTPS à un site web. Identifiez : (a) le handshake TCP (SYN, SYN-ACK, ACK), (b) le handshake TLS (ClientHello, ServerHello), (c) la cipher suite négociée. Expliquez pourquoi le contenu des échanges est illisible après le handshake.

Voir la solution 
# Capture avec tcpdump (30 secondes sur l'interface eth0)
sudo tcpdump -i eth0 -w capture.pcap -c 100 port 443

# Ouvrir avec Wireshark et appliquer le filtre :
# tcp.port == 443

# (a) Handshake TCP (3 premiers paquets) :
# Paquet 1 : Client → Serveur  [SYN]        seq=0
# Paquet 2 : Serveur → Client  [SYN, ACK]   seq=0 ack=1
# Paquet 3 : Client → Serveur  [ACK]        seq=1 ack=1

# (b) Handshake TLS (dans les paquets suivants) :
# ClientHello : versions TLS supportées, cipher suites proposées,
#               extensions (SNI avec le nom de domaine)
# ServerHello : version TLS choisie, cipher suite sélectionnée,
#               certificat du serveur

# (c) Cipher suite typique (TLS 1.3) :
# TLS_AES_256_GCM_SHA384
# = AES-256-GCM pour le chiffrement, SHA-384 pour le hachage

# Pourquoi le contenu est illisible :
# Après le handshake, toutes les données sont chiffrées avec AES-GCM
# en utilisant les clés de session dérivées de l'échange Diffie-Hellman.
# Seuls le client et le serveur possèdent ces clés.

Exercice 3 — Détection d'ARP spoofing

Écrivez un script Python qui surveille les réponses ARP sur le réseau local et détecte les tentatives d'ARP spoofing (une même adresse IP associée à deux adresses MAC différentes). Utilisez la bibliothèque scapy.

Voir la solution 
from scapy.all import sniff, ARP
from collections import defaultdict
import datetime

# Table ARP : IP → MAC
table_arp = {}
alertes = []

def detecter_arp_spoofing(paquet):
    if paquet.haslayer(ARP) and paquet[ARP].op == 2:  # ARP reply
        ip = paquet[ARP].psrc
        mac = paquet[ARP].hwsrc

        if ip in table_arp:
            if table_arp[ip] != mac:
                alerte = (
                    f"[ALERTE] ARP Spoofing détecté !\n"
                    f"  IP : {ip}\n"
                    f"  MAC précédente : {table_arp[ip]}\n"
                    f"  MAC actuelle   : {mac}\n"
                    f"  Heure : {datetime.datetime.now()}"
                )
                print(alerte)
                alertes.append(alerte)
        else:
            print(f"[INFO] Nouvelle entrée ARP : {ip} → {mac}")

        table_arp[ip] = mac

print("Surveillance ARP en cours... (Ctrl+C pour arrêter)")
sniff(filter="arp", prn=detecter_arp_spoofing, store=0)

# Contre-mesures ARP spoofing :
# 1. Entrées ARP statiques pour les serveurs critiques
# 2. Dynamic ARP Inspection (DAI) sur les switchs
# 3. 802.1X pour l'authentification réseau

Exercice 5 — Scan de ports et durcissement

En utilisant Nmap, scannez un serveur de test (vous pouvez utiliser scanme.nmap.org qui autorise les scans). Listez les ports ouverts, identifiez les services et leurs versions. Pour chaque service trouvé, proposez une mesure de durcissement.

Voir la solution 
# Scan avec détection de version
nmap -sV -sC scanme.nmap.org

# Résultat typique :
# PORT      STATE  SERVICE     VERSION
# 22/tcp    open   ssh         OpenSSH 8.9p1
# 80/tcp    open   http        Apache httpd 2.4.52
# 9929/tcp  open   nping-echo  Nping echo
# 31337/tcp open   tcpwrapped

# Mesures de durcissement par service :

# SSH (port 22) :
# → Désactiver l'authentification par mot de passe (clés uniquement)
# → Changer le port par défaut (ex: 2222)
# → Limiter les utilisateurs autorisés (AllowUsers)
# → Installer fail2ban pour bloquer les tentatives de brute force
# → Désactiver le login root (PermitRootLogin no)

# HTTP (port 80) :
# → Rediriger vers HTTPS (port 443)
# → Masquer la version du serveur (ServerTokens Prod)
# → Activer les en-têtes de sécurité (CSP, HSTS, X-Frame-Options)
# → Désactiver les modules Apache inutiles
# → Mettre à jour vers la dernière version

# Ports non nécessaires (9929, 31337) :
# → Fermer avec le pare-feu s'ils ne sont pas utilisés
# → iptables -A INPUT -p tcp --dport 9929 -j DROP
# → iptables -A INPUT -p tcp --dport 31337 -j DROP

# Général :
# → Activer un IDS (Suricata) pour surveiller le trafic
# → Journaliser tous les accès (rsyslog → SIEM)
